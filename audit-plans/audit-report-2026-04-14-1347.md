## Findings by Category

### 1. Security

#### High — Google API key transmitted in URL query string

**File**: `src/routes/settings.js:479`

```js
if (provider === 'google') {
  const res = await fetch(
    `https://generativelanguage.googleapis.com/v1beta/models?key=${apiKey}`
  );
```

Query-string keys appear in: Cloudflare request logs, any intermediate proxy logs, browser history (if ever served through one), Referer headers if the response body triggers any further request, and any server-side APM tooling that logs URLs. The Anthropic and OpenAI branches above use `Authorization`/`x-api-key` headers correctly.

**Fix**: Google's generative-language API accepts the key via the `x-goog-api-key` header:
```js
const res = await fetch('https://generativelanguage.googleapis.com/v1beta/models', {
  headers: { 'x-goog-api-key': apiKey }
});
```

#### High — `decryptSensitiveData` accepts colon-less plaintext silently (partial fix)

**File**: `src/utils/crypto.js:583-585`

```js
if (!encryptedData.includes(':')) {
  return encryptedData;
}
```

The 2026-04-09 fix added an `enc:` prefix to new ciphertext (good — detects tampering on properly-formatted writes) but left the plaintext fallback. Any field that ends up colon-less (bug in encrypt path, partial migration, manual DB edit) is returned as-is with no alarm. At minimum this should log a warning to Sentry on every plaintext read, so ops knows a field escaped encryption.

**Fix**: Add `console.warn` + Sentry capture when the fallback fires, including the field length and call site (stack). Then schedule a migration to fail-closed after confirming zero plaintext reads in production.

#### High — Error-message leakage in book import confirm batch

**File**: `src/routes/books.js:1909` (approx — agent reports this location)

```js
} catch (error) {
  console.error(`Batch ${Math.floor(i / BATCH_SIZE) + 1} failed:`, error.message);
  batch.forEach((b) => b.onError(error.message));
}
```

`error.message` from D1 (UNIQUE constraint violation, FK failure, table hints) is forwarded to the client. The endpoint requires `requireAdmin()` so exposure is authenticated-only, but any admin from any organization can now read schema-tier error detail.

**Fix**: `batch.forEach((b) => b.onError('Import batch failed. Please contact support.'));`

#### High — Platform AI key resolution surfaces decrypt errors

**Files**: `src/routes/books.js:669, 702` (approx, per agent)

Decryption failures in the AI resolution path propagate raw error text. Also applies to any org-level `org_ai_config` decrypt failure.

**Fix**: Wrap decrypt calls in try/catch; return `{ available: false, reason: 'config_error' }` to callers; log the real error server-side.

#### Medium — `tenantMiddleware` hits D1 on every authenticated request

**File**: `src/middleware/tenant.js:106-109`

```js
const org = await db
  .prepare('SELECT id, is_active, subscription_status FROM organizations WHERE id = ?')
  .bind(targetOrgId)
  .first();
```

No caching layer. Every API call from every client pays this round-trip. Owners doing org-switching pay an *additional* query (user role lookup at lines 92-95). At 30 schools × 10 concurrent users × 50 req/min = 15k unnecessary D1 reads/min.

**Fix**: Cache `(is_active, subscription_status)` in KV with 5-10 min TTL keyed by org ID. Invalidate from the Stripe webhook when status transitions and from admin org-deactivation routes. Keep the current query as the source of truth on cache miss.

#### Medium — Badge engine `genres` query spans all orgs (design-accepted)

**File**: `src/utils/badgeEngine.js:71`

```js
const genresResult = await db.prepare('SELECT id, name FROM genres').all();
```

Comment now calls this intentional ("genres are global, not org-scoped"), and the genres table does appear to be shared. But orgs can add genres via `POST /api/genres` — so a genre named by school A is visible to badge classification at school B. Risk: cross-tenant information disclosure via badge category routing, and badge stats poisoning if a school adds adversarial genre names.

**Fix**: Either (a) make genres truly platform-scoped (remove per-org write path), or (b) scope the query to the student's organization. Option (b) is safer.

#### Medium — Stripe webhook swallows trial-ending email failures

#### Low — `classes.js` soft-delete UPDATE missing org-scope (defense-in-depth)

**File**: `src/routes/classes.js:355-358`

### 2. Bugs & Data Integrity

#### High — HomeReadingRegister first useEffect still lacks AbortController

**File**: `src/components/sessions/HomeReadingRegister.js:194-214`

```js
useEffect(() => {
  if (!effectiveClassId) { setClassSessions([]); return; }
  if (!fetchStartDateISO || !endDateISO) return;
  setSessionsLoading(true);
  fetchWithAuth(`/api/students/sessions?classId=…&startDate=…&endDate=…`)
    .then(r => r.ok ? r.json() : [])
    .then(sessions => { setClassSessions(sessions); setSessionsLoading(false); })
    .catch(() => { setClassSessions([]); setSessionsLoading(false); });
}, [effectiveClassId, fetchStartDateISO, endDateISO, fetchWithAuth]);
```

The *second* useEffect (lines 228-258, student history) uses AbortController correctly. This one (the main class sessions fetcher) does not. Rapidly switching class or date range produces racing responses that can overwrite fresher data with staler data. The prior audit's item #10 flagged HomeReadingRegister but this particular effect was missed.

**Fix**: Mirror the pattern at line 233 — `const controller = new AbortController();`, pass `signal`, check `error.name === 'AbortError'`, return cleanup.

#### Medium — `refreshSessions` swallows all errors

**File**: `src/components/sessions/HomeReadingRegister.js:217-225`

```js
return fetchWithAuth(...)
  .then(r => r.ok ? r.json() : [])
  .then(setClassSessions)
  .catch(() => {});
```

After mutation (add/delete session), if the refetch fails, the UI silently shows stale state. No toast, no retry, no Sentry.

**Fix**: `.catch((err) => { console.error('[HomeReading] refresh failed', err); setSessionsError(err.message); })` with a small `<Alert>` banner above the register.

#### Medium — `orgPurge` has no transactional guarantee across 26 tables

**File**: `src/services/orgPurge.js:94-143`

```js
for (const { table, where } of DELETE_ORDER) {
  try {
    await db.prepare(`DELETE FROM ${table} WHERE ${where}`).bind(orgId).run();
  } catch (error) { errors.push(…); }
}
// …then anonymise org row regardless of errors
```

If any mid-purge delete fails (row lock, FK failure due to a new table added to schema but not DELETE_ORDER, D1 transient failure), the loop continues. The final `UPDATE organizations SET name='Deleted Organisation'…` runs unconditionally. Result: an org tombstoned with partial data still in one or more tables — a GDPR failure mode.

D1 doesn't support traditional BEGIN/COMMIT across prepare/run, but it does support atomic `db.batch()` within the 100-statement cap. A 26-delete cascade is easily under the cap if rewritten to pre-build the statements array.

**Fix**: Rewrite to build all 26 DELETE statements into a single `db.batch([...])` call. Also fail loudly (throw) rather than continuing past an error — an incomplete purge must be re-run, not papered over.

#### Medium — `wondeSync` full sync has no org-level lock

**File**: `src/services/wondeSync.js:200-286` (per agent)

If the nightly cron and a manual "Sync Now" from the admin UI overlap, both run `runFullSync` against the same org. The DELETE phase of one can race with the INSERT phase of the other, briefly leaving the org with mixed or missing student/class data.

**Fix**: Acquire a lightweight lock before sync — either (a) an `is_syncing = 1` flag on the org row checked via `UPDATE ... WHERE is_syncing = 0` (returns rows-affected), or (b) a KV-based mutex with short TTL. Release in finally.

#### Medium — PlatformSettings key deletion lacks confirmation dialog

**File**: `src/components/PlatformSettings.js:143-173`

```js
const handleDeleteKey = async (provider) => {
  setDeletingProvider(provider);
  try {
    const response = await fetchWithAuth(`${API_URL}/settings/platform-ai/${provider}`, {
      method: 'DELETE',
    });
    // …
```

No confirmation. A misclick on "Remove" deletes the platform-level AI key, which may be the only key serving many schools (schools without their own `org_ai_config` fall back to platform). Every AI recommendation call for those schools fails until the key is re-entered.

**Fix**: Add a confirmation Dialog: "This will disable AI features for all schools using the platform key. Type DELETE to confirm." This matches the destructiveness of org-delete flows elsewhere in the app.

### 3. Performance & Scalability

#### High — Badge evaluation cron iterates all orgs × all students sequentially

**File**: `src/worker.js:617-643`

Current pattern:
```js
for (const org of orgs.results || []) {
  const students = await db.prepare(`SELECT DISTINCT s.id … FROM students s INNER JOIN reading_sessions rs … WHERE s.organization_id = ?`).bind(org.id).all();
  for (const student of students.results || []) {
    await recalculateStats(db, student.id, org.id);
    const rtBadges = await evaluateRealTime(db, student.id, org.id, student.year_group);
    const batchBadges = await evaluateBatch(db, student.id, org.id, student.year_group);
  }
}
```

Each `recalculateStats` + `evaluateRealTime` + `evaluateBatch` issues multiple D1 queries per student. At 30 orgs × 300 students/org, that's ~9k students × ~10 queries = 90k D1 reads/writes per nightly run. Scheduled Workers have a 30s CPU limit. At current scale it likely completes, but there's no observability on elapsed time and no chunking plan. Will break silently at 100+ orgs.

**Fix**: Phase 1 — add elapsed-time logging so we know how close to the edge we are. Phase 2 — chunk orgs into groups of 5-10, use `scheduled` events or the `Queue` binding to fan out. Phase 3 — combine `recalculateStats`/`evaluateRealTime`/`evaluateBatch` into fewer queries per student.

#### Medium — Badge engine author loop fires N subqueries per student (Series Finisher)

**File**: `src/utils/badgeEngine.js:352-367` (per agent)

For each distinct author a student has read, a separate `SELECT COUNT(DISTINCT book_id)` runs. 20 authors per student × 9k students = 180k queries in the cron window.

**Fix**: One query with `GROUP BY author` and filter in memory.

#### Medium — Class-goals recalc runs 6 separate aggregation queries per class

**File**: `src/utils/classGoalsEngine.js:144-246` (per agent)

Sessions, books, genres, reading_days, readers, badges — each a separate query. 50 classes × 6 queries × 30 orgs = 9k queries per cron.

**Fix**: Combine into a single query with subqueries in the SELECT list (D1 supports SQLite subqueries).

#### Medium — `students` GET uses SELECT * with correlated subqueries

**File**: `src/routes/students.js:243-250` (per agent)

```js
SELECT s.*, c.name as class_name, b.title as current_book_title, …,
  (SELECT COUNT(*) FROM reading_sessions rs WHERE rs.student_id = s.id) as total_session_count,
  (SELECT COUNT(*) FROM student_badges sb WHERE sb.student_id = s.id) as badge_count
```

Correlated subqueries fire once per row. 1000 students = 2000 subquery executions for the primary list endpoint.

**Fix**: `LEFT JOIN` to pre-aggregated `student_reading_stats` + `student_badges` counts. Avoid SELECT * — list needed columns.

#### Medium — Missing compound index on `reading_sessions`

Current indexes (based on migrations review) cover `student_id + session_date` but not `organization_id`. Queries in badgeEngine and classGoalsEngine filter by `organization_id` first in joins; the query planner can't use the current index directly for the composite path.

**Fix**: Add migration with `CREATE INDEX idx_reading_sessions_org_student_date ON reading_sessions(organization_id, student_id, session_date DESC);`

#### Low — KV caches grow unbounded in long-lived namespaces

**Files**: `src/utils/recommendationCache.js`, `src/utils/isbnLookup.js`

Both use `expirationTtl` which is correct for eventual eviction, but on very active namespaces the working set can exceed free-tier KV quotas.

**Fix**: Monitor via Cloudflare dashboard. Consider reducing the recommendations cache TTL from 7 days to 1-2 days — AI models change quickly and stale recommendations are cheaper to recompute than stale book metadata.

#### Low — `metadataService` processes books serially with 1500ms delays

**File**: `src/services/metadataService.js:106-159`

Worker cron timeout (30s) + 1500ms per book = ~19 books per run. Large orgs backfilling 1000+ books take many weeks. Could parallelise within each provider's rate limit.

**Fix**: Worth revisiting if metadata backfill becomes a blocker; not urgent today.
