## Findings by Category

### A. Performance & Cost (Cloudflare D1/Workers)

#### H1 — Still no bulk session endpoint: register multi-day entry fires up to 28 sequential POSTs × ~29 D1 queries each
**Severity: High** (cost + latency; residue of prior H5) | **Files**: `src/components/sessions/HomeReadingRegister.js:543–606` (loop; marker-day branch at `:573–593` fires **two** POSTs per iteration), `src/routes/students/sessions.js:165–350`, `src/routes/parent.js:235–383`

The server side was improved (side-effects parallelised, commit b77cfde), but each session POST still costs **~29 D1 queries** on the response path (core batch ~3 + streak 2 + stats 3 + class-goal ~15 + band 4 + badges 2). The client still loops sequential awaited POSTs per backfill day. A 14-day catch-up with markers ≈ 28 POSTs ≈ **800+ D1 queries for one tap**; a class-wide 5-day catch-up (20 students) ≈ ~2,900. A Sunday-evening parent burst at 10k students remains the costliest realistic scenario.

**Fix** (do together with B-H2 below): add `POST /api/students/:id/sessions/bulk` accepting `[{date, bookId, …}]`, insert all rows in one batch, run the side-effect chain **once** at the end (streak/stats/band/goals are all recompute-from-state, so once-per-student is correct), and have `handleQuickRecord` call it. ~5×29 → ~1×40 queries per student.

#### M3 — Class-goal update still rescans the full academic year on every session write (carryover)
**Severity: Medium** | **File**: `src/utils/classGoalsEngine.js:254–296` → `recalculateClassGoalProgress:131–199`

Every teacher/parent session POST runs the 6-subquery full-year aggregate (one with `JOIN json_each(b.genre_ids)`). Mitigations since last cycle: early-exit when the class has no goals (`:289`), and it runs inside the parallel side-effect block. Still the single largest contributor (~15 of the ~29 per-write queries). **Fix**: incremental counter updates (+1 sessions, +pages, conditional uniques) with the nightly drift cron as reconciliation — or drop the per-write recalc entirely and rely on the cron + optimistic client increment.

#### L4 — Org-list correlated subqueries (carryover, downgraded)
**Severity: Low** (was Medium) | **File**: `src/routes/organization.js:179–191`
Still 4 correlated subqueries per row, but the supporting indexes landed (`idx_students_org_active` etc.) and it's owner-only over a small org set. **Fix when convenient**: LEFT JOIN derived tables.

#### L5 — Demo-reset badge phase sequential and un-hoisted (carryover)
**Severity: Low** (bounded: 24 demo students, hourly) | **File**: `src/services/demoReset.js:220–222`
Per-student `recalculateStats` + `evaluateRealTime` + `evaluateBatch`, no hoisted genre map. **Fix**: use the batched `processBadgesForOrg`.

#### L6 / L7 / L8 — Carryover hygiene (all re-verified present)
- **L6**: `rate_limits` cleanup is `Math.random() < 0.01` per write (`tenant.js:457`) — fold into the existing daily cron and delete the probabilistic branch.
- **L7**: parent book search hits OpenLibrary per query, uncached (`parent.js:475`) — KV-cache 24h.
- **L8**: `generateUniqueSlug` awaited per school inside wonde sync-all loop (`wondeAdmin.js:135`) — pre-resolve collisions in one query.

#### L9 — ISBN verification fetches sequentially (new)
**Severity: Low** (owner-only, capped at 20) | **File**: `src/routes/books/duplicates.js:150–160`
20 ISBNs × 5s timeout worst case ≈ 100s request. **Fix**: `Promise.all` with concurrency ~5; optionally a per-owner rate limit on the endpoint.

**Verified clean (performance)**: band recompute is cheap — pure `computeBandIndex` from a count, no `COUNT(*)` per write; one bounded `SELECT notes … WHERE session_date >= yearStart` per student per write (`_shared.js:294`). Band/streak settings are KV-cached 1h (`_shared.js:201–267`). Observation parsing is a pure 6-column map. Duplicate detection is GROUP-BY + in-JS union-find clustering (no O(n²) SQL) with count fetches chunked at 400; merge is a single atomic batch capped at 50.

### B. Code Quality & Maintainability (emphasis area)

#### H2 — Session side-effect chain duplicated between teacher and parent routes
**Severity: High** (maintainability) | **Files**: `src/routes/students/sessions.js:287–317` vs `src/routes/parent.js:348–359`

Both routes hand-roll the same chain — `runSafe` wrapper + `Promise.all([updateStudentStreak, recalculateStats, updateClassGoalOnSession, updateStudentBand])` then `evaluateRealTime` — with slightly different option handling. The reading-bands feature had to be wired in **twice** (and was — correctly, this time). The next side-effect (or ordering fix) must also be made twice, and a missed edit produces silent teacher/parent divergence — exactly the class of bug (H2-2026-05-29) this codebase just fixed.

**Fix**: extract `runSessionSideEffects(db, env, { studentId, organizationId, yearGroup, isMarkerSession })` → `{ newBadges, completedGoals, bandResult, streakData }` into `src/routes/students/_shared.js`; call from both routes. Natural companion to the bulk endpoint (A-H1).

#### M4 — Three string-matching utilities with two different similarity algorithms
**Severity: Medium** | **Files**: `src/utils/stringMatching.js`, `src/utils/titleMatching.js`, `src/utils/bookDedup.js`, plus a local `normalizeTitle()` re-implementation in `src/components/books/bookImportUtils.js`

`stringMatching` (Levenshtein, 0.85 threshold — import dedup) and `titleMatching` (bigram-Jaccard + word overlap + substring, 0.3-ish thresholds — metadata-provider ranking) answer "is this the same book?" with different algorithms; `bookImportUtils` redefines normalisation locally instead of importing. `bookDedup` (new) does it right — it reuses `stringMatching`. A title pair can match in the import wizard and not in provider ranking. **Fix**: make `stringMatching.js` the single home for normalisation + both similarity strategies (named exports `similarityLevenshtein` / `similarityFuzzy`), fold `titleMatching` into it, delete the local copy in `bookImportUtils`, document when to use which.

#### M5 — God files: `auth.js` (1,051) and `settings.js` (791) mix 4+ concerns each
**Severity: Medium** | **Files**: `src/routes/auth.js`, `src/routes/settings.js`; honourable mentions `BookRecommendations.js` (1,258), `SessionForm.js` (1,157), `HomeReadingRegister.js` (1,082)

The codebase already has the right pattern — `students/` and `books/` sub-routers — these two just never got it. **Fix**: split on the existing convention (`auth/{login,password,register}.js`, `settings/{org,ai}.js`). ### C. Security

#### M1 — Covers route never got a rate-limit bucket: per-key buckets still defeat the aggregate cap
**Severity: Medium** (residue of prior H3 — the half that motivated the High is the half left unfixed) | **File**: `src/routes/covers.js:17`
```js
coversRouter.use('*', rateLimit(60, 60000));   // no bucket → falls back to c.req.path
```
The bucket mechanism exists and the parent routes use it (`parent:view|sessions|books` — verified), but covers still keys on the concrete path, so every distinct `/api/covers/:type/:key` gets its own 60/min counter — unbounded R2 reads + external provider fan-out for an unauthenticated attacker iterating ISBNs. **Fix**: `rateLimit(60, 60000, 'covers')` — one line. **Related footgun (L1)**: `bucket` is optional; any future parameterised public route silently regresses. Consider logging a warning (or throwing in dev) when `rateLimit` runs on a path containing a param segment without an explicit bucket.

**Severity: Low** | **File**: `src/routes/books/duplicates.js:169` (`auditLog('merge','books')`)
A merge repoints `reading_sessions`/`current_book_id` across **all** orgs holding the merged books, but the audit row carries only the owner's JWT org. **Fix**: include merged book ids + affected org ids (from the `org_book_selections` GROUP BY already computed) in the audit detail.

#### L3 — Unknown classId on sessions GET returns `[]` rather than 404
**Severity: Low** (cosmetic; isolation verified intact) | **File**: `src/routes/students/sessions.js:64–75`
Org scoping makes foreign ids harmless (verified — see Refuted), but validating classId existence would aid client debugging and remove the ambiguity that misled the review agent.
