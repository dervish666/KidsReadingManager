## Executive Summary

The codebase is in strong shape for performance. The recent architecture improvements (context splitting, route decomposition, genreNameMap hoist, org-status cache TTL tuning) show a mature approach to optimisation. The student list — the hottest endpoint — already uses collapsed aggregation queries and 30s browser caching.

That said, there are **real wins still on the table**, mostly in two areas:

1. **Backend query consolidation**: The badge cron does 4–6 D1 queries per student per nightly run, with per-badge INSERTs instead of batched writes. The stats endpoint fetches individual session rows for aggregations that SQL could handle. The org list uses 4 correlated subqueries per row.

2. **Caching gaps**: Most API endpoints lack Cache-Control headers, static assets rely on Cloudflare defaults without explicit immutable headers for hashed files, and the stats endpoint does expensive aggregation with no caching layer.

## Findings by Category

### 1. Database Query Performance

#### H1 — Badge cron: per-student query overhead + unbatched badge INSERT
**Severity: High** | **Files**: `src/utils/badgeEngine.js:360–418`, `src/worker.js:602–720`

Each student processed during the nightly badge cron triggers 4–6 D1 queries:
- `recalculateStats()`: 2 queries (sessions + books)
- `evaluateRealTime()`: 2 queries (earned badges + reading stats)
- `evaluateBatch()`: 3 queries (earned badges + sessions + author book counts)

Additionally, earned badges are INSERTed one at a time (line 412–418):
```js
await db.prepare(
  `INSERT INTO student_badges (id, student_id, organization_id, badge_id, tier)
   VALUES (?, ?, ?, ?, ?)`
).bind(...).run();
```

At 200 students per cron invocation × 5 queries = 1,000 D1 subrequests — right at the Workers paid-tier limit. The 22s budget timer provides a safety net, but this is the binding constraint on how many students can be processed per nightly run.

**Fix**: (a) Batch-insert earned badges — collect all new badges per student and insert in a single `db.batch()` call. (b) Prefetch author book counts once per org (like genreNameMap) instead of per-student. (c) Consider combining the `evaluateRealTime` and `evaluateBatch` earned-badge lookups into a single query.

#### H2 — Stats endpoint fetches raw rows for SQL-capable aggregation
**Severity: High** | **File**: `src/routes/students/stats.js:94–149`

GET `/api/students/stats` fetches individual `reading_sessions` rows (chunked by 90 student IDs), then loops through all rows in-memory to compute location distribution, day-of-week counts, weekly activity, and most-read books.

For a 500-student org with 50 sessions each = 25,000 rows fetched, parsed to JSON, and iterated. Most of these aggregations are expressible in SQL:
```sql
SELECT
  COUNT(*) as total,
  SUM(CASE WHEN location = 'home' THEN 1 ELSE 0 END) as home_count,
  SUM(CASE WHEN session_date >= ? THEN 1 ELSE 0 END) as this_week
FROM reading_sessions WHERE student_id IN (...)
```

The per-student `studentLastReadMap` tracking and `mostReadBooks` ranking could remain in JS, but the bulk aggregation should move to SQL.

**Impact**: Saves ~80% D1 row reads and response serialization overhead on this endpoint.

#### H3 — Organization list: 4 correlated subqueries per row
**Severity: High** | **File**: `src/routes/organization.js:179–191`

The owner-only `/api/organization/all` endpoint runs 4 correlated subqueries per organization row:
```sql
SELECT o.*,
  (SELECT COUNT(*) FROM students s WHERE s.organization_id = o.id AND s.is_active = 1) as student_count,
  (SELECT COUNT(*) FROM classes c WHERE c.organization_id = o.id AND c.is_active = 1) as class_count,
  (SELECT wsl.error_message FROM wonde_sync_log wsl WHERE wsl.organization_id = o.id ORDER BY wsl.started_at DESC LIMIT 1) as last_sync_error,
  (SELECT COUNT(*) FROM org_ai_config WHERE organization_id = o.id AND api_key_encrypted IS NOT NULL) as has_ai_key
FROM organizations o ... LIMIT 50 OFFSET ?
```

At 50 rows per page × 4 subqueries = 200 correlated scans. With 1,000+ students across orgs, each `COUNT(*)` subquery on `students` scans the table.

**Fix**: Replace with LEFT JOIN + GROUP BY, or move counts to a separate pre-aggregated view/cache.

#### M1 — Unbatched wondeAdmin school upserts
**Severity: Medium** | **File**: `src/routes/wondeAdmin.js:107–146`

POST `/api/wonde/schools` iterates over schools and executes individual INSERT/UPDATE per school. Should use `db.batch()` with 100-statement chunks.

**Impact**: Low in practice (typically 1–5 schools per sync), but architecturally inconsistent with the batching discipline used elsewhere (e.g., `wondeSync.js` correctly batches).

#### M2 — Demo reset: sequential deletes
**Severity: Medium** | **File**: `src/services/demoReset.js:128–135`

Hourly demo reset runs ~20 DELETE queries sequentially. FK-dependent deletes must stay sequential, but independent deletes (e.g., `student_preferences`, `org_book_selections`, `student_badges`) could batch together.

**Impact**: ~1–2s per hourly reset. Low urgency but easy win.

#### M3 — Class goals drift correction: no budget guard
**Severity: Medium** | **File**: `src/worker.js:684–711`

The class goals recalculation loop (after badge cron) iterates all orgs and all classes without a `Date.now() > deadlineMs` check. Unlike the badge cron which has cursor-based resumption, this loop runs unbounded within the same 30s cron invocation.

```js
for (const org of orgs.results || []) {
  const classes = await db.prepare('SELECT id FROM classes WHERE ...').bind(org.id).all();
  for (const cls of classes.results || []) {
    await recalculateClassGoalProgress(db, cls.id, org.id, startDate, endDate, term);
  }
}
```

At 100 orgs × 10 classes × 1 query each = 1,000+ queries, this could exhaust the remaining CPU budget (badge cron uses 22s of 30s).

**Fix**: Add a deadline check before each class, or move to a separate cron trigger.

#### L1 — No composite index for badge sort
**Severity: Low** | **File**: `src/routes/badges.js:41`

`SELECT * FROM student_badges WHERE student_id = ? ORDER BY earned_at DESC` — no composite index on `(student_id, earned_at DESC)`. Low impact (typically <100 badges per student).

### 2. Caching Strategies

#### H4 — Missing Cache-Control headers on hot-path API endpoints
**Severity: High** | **Files**: `src/routes/books.js`, `src/routes/students/stats.js`, `src/routes/students/sessions.js`

Only three endpoints set explicit Cache-Control headers:
- `/api/students` list → `private, max-age=30` ✓
- `/api/genres` → `private, max-age=60` (via worker.js fallback) ✓
- `/api/covers/*` → `public, max-age=2592000` (hit) / `max-age=3600` (miss) ✓

All other GET endpoints (books list, books search, student detail, stats, sessions, badges) return **no Cache-Control header**, meaning browsers treat them as uncacheable. The data context reloads these on every org switch and periodically — short-lived caching (30–60s) on read-heavy endpoints would reduce D1 load significantly.

**Priority endpoints for caching**:
| Endpoint | Suggested TTL | Rationale |
|----------|--------------|-----------|
| `/api/students/stats` | 60–120s | Most expensive aggregation; rarely stale within 2 min |
| `/api/books?all=true` | 60s | Used by autocomplete on every DataContext reload |
| `/api/books/search` | 30s | FTS5 search on every keystroke |
| `/api/badges/:studentId` | 30s | Badge collection rarely changes mid-session |

#### M4 — Static assets: no explicit immutable Cache-Control
**Severity: Medium** | **File**: `src/worker.js:398–409`

Static assets are served via `env.ASSETS.fetch(request)` with security headers added, but no Cache-Control header. Rsbuild generates content-hashed filenames (e.g., `lib-mui.77b63abb.js`), which should get `Cache-Control: immutable, max-age=31536000`. Currently relies on Cloudflare's default behaviour.

**Fix**: Add conditional Cache-Control based on path pattern:
- Hashed JS/CSS files → `immutable, max-age=31536000, public`
- `index.html` / `/` → `no-cache, must-revalidate, public`

#### M5 — Recommendation cache: studentId not included in cache inputs
**Severity: Medium** | **File**: `src/routes/books/recommendations.js:322–329`, `src/utils/recommendationCache.js:22–31`

The `cacheInputs` object passed to `getCachedRecommendations()` does not include `studentId`:
```js
const cacheInputs = {
  readingLevelMin, readingLevelMax, genres, focusMode, recentBookIds, provider
};
```

Meanwhile `generateCacheKey()` includes `studentId: inputs.studentId || ''`, which defaults to `''` for all calls. Two students with identical reading profiles across different orgs would share a cache entry.

**Not a data leak** — cached responses contain only book titles/reasons (not PII), and a library cross-check (line 342–359) verifies suggestions against the current org's library. But it's architecturally sloppy: add `organizationId` to the key for correctness, since the AI prompt includes org-specific book candidates.

#### L2 — Streak settings cache: no explicit invalidation
**Severity: Low** | **File**: `src/routes/students/_shared.js:92–144`

`getOrgStreakSettings()` caches in KV with a 1-hour TTL but has no explicit invalidation when settings are changed. Changes take up to 1 hour to propagate. Acceptable for streak thresholds, but worth noting.

### 3. Frontend Performance

#### M6 — html5-qrcode eagerly imported in a commonly-loaded chunk
**Severity: Medium** | **File**: `src/components/books/BarcodeScanner.js:11`

```js
import { Html5Qrcode, Html5QrcodeSupportedFormats } from 'html5-qrcode';
```

BarcodeScanner is imported statically by ScanBookFlow, which is imported by BookAutocomplete (used in SessionForm, FullReadingView, QuickReadingView). Although these parent pages are lazy-loaded, html5-qrcode (~50KB) is bundled into their chunks even though barcode scanning is a low-frequency feature.

**Fix**: Convert to dynamic import:
```js
const { Html5Qrcode, Html5QrcodeSupportedFormats } = await import('html5-qrcode');
```

#### L3 — No list virtualization
**Severity: Low** | **Mitigated by**: Pagination (25/page in StudentList)

No react-window/react-virtualized dependency. StudentList paginates at 25 rows, so this isn't a problem today. Only becomes relevant if any view renders 500+ rows without pagination (currently none do).

#### L4 — No prefetch hints for lazy-loaded chunks
**Severity: Low** | **File**: `rsbuild.config.mjs`

Lazy-loaded pages (Stats, BookManager, HomeReadingRegister) could benefit from `<link rel="prefetch">` hints after initial render. Would improve perceived tab-switch performance. Nice-to-have.

**Strengths noted**: Context providers properly memoized with `useMemo`/`useCallback`. BookCover uses IntersectionObserver for lazy image loading. MUI imports are tree-shakeable. React.memo applied to high-frequency components (BookCover, StudentTable, StudentCard). Route-level code splitting is comprehensive.

### 4. Build Pipeline & Developer Experience

#### H5 — CI pipeline missing lint check
**Severity: High** | **File**: `.github/workflows/build.yml`

The CI pipeline runs `prettier --check` and `npm run test:coverage` but does **not** run `npm run lint`. ESLint was adopted at v3.57.4 with `no-undef` (error) and `react-hooks/rules-of-hooks` (error) — these catch real bugs. Without lint in CI, a PR could pass CI with ESLint errors.

**Fix**: Add `npm run lint` step after formatting check, before build.

#### M7 — No bundle size monitoring
**Severity: Medium** | **File**: `package.json`

No bundle analyzer configured. Adding `@rsbuild/plugin-bundle-analyzer` (or equivalent) would provide visibility into chunk sizes during development and help catch accidental bundle bloat. Could also add a CI step that fails on bundle size regression.

#### L5 — No `npm run dev:debug` convenience script
**Severity: Low** | **File**: `package.json`

Debugging the Worker requires knowledge of Wrangler CLI flags. A convenience script would lower the barrier for contributors.

**Strengths noted**: Build config is mature (chunk splitting, source maps, minification). CI caches npm deps. E2E tests run on PRs only (cost-conscious). Prettier formatting enforced in CI. `concurrently` provides good dev feedback loop.

## Verification Downgrades

1. **Recommendation cache cross-tenant leakage** — Agent rated Critical. **Downgraded to Medium.** Student IDs are UUIDs (globally unique, `generateId()` in helpers.js uses `crypto.getRandomValues`). The `studentId` field isn't passed in cache inputs (defaults to `''`), making the cache profile-based rather than student-based. Cached responses are book titles only (not PII). Library cross-check on cache hit (line 342–359) validates against current org's library. This is architecturally sloppy but not a data leak.

2. **Badge cron subrequest budget "200,000 subrequests"** — Agent rated Critical. **Downgraded to High.** The agent multiplied across all orgs (100 × 500 × 4 = 200,000) but the entire cron runs in a single invocation with a 22s budget timer and cursor-based resumption. The real constraint is ~200 students per invocation × 5 queries = 1,000 subrequests (Workers paid limit). The budget timer naturally caps this before the limit is hit. Still worth optimizing per-student query count.

3. **Static asset caching "no Cache-Control"** — Agent rated Medium. **Confirmed Medium.** Cloudflare Workers Assets does handle basic caching via CDN defaults, but explicit `immutable` headers for content-hashed files would improve cache hit rates for returning users.

4. **CI "no linting"** — Agent claimed no lint step. **Confirmed.** `prettier --check` runs but `npm run lint` does not. Since ESLint catches real bugs (`no-undef`, `rules-of-hooks`), this is High.
