## Executive Summary

The codebase is in strong shape and hardening has stuck: the full launch-blocker set from the 2026-04-17 pen-test (C1–C4, H5/H6/H8/H11, H10/M14–M16/M18) is verified fixed, `orgStatusCache` is correctly wired through Stripe webhook, org deactivation, and purge, and the test suite went from ~600 to 1948 tests with full green. This cycle — 4 parallel deep-dive agents plus direct verification — found **0 Critical, 4 High, 9 Medium, 6 Low** new or still-present findings. Several agent-proposed Critical/High ratings were downgraded after verification (see Verification Downgrades).

**Most significant discoveries this run**:

1. **Session POST hot path does 6+ sequential D1 writes with no try/catch and no batch** (High, new) — `src/routes/students.js:1250-1406`. A mid-sequence failure leaves the student with stale `current_book_id`/`last_read_date` while the session row exists. Not a corruption risk per-se (next session self-heals), but worth stronger guarantees.
2. **`classGoalsEngine.recalculateClassGoalProgress()` runs 6 separate aggregation queries on every goals GET** (High, new) — `src/routes/classes.js:429`. Teachers viewing the classroom projection view will trigger this repeatedly; 6 D1 queries per dashboard poll is expensive at scale.
3. **`/api/students` list uses SELECT \* with two correlated subqueries** (High, carryover from 2026-04-14) — verified still present at `src/routes/students.js:243-250`. Scales linearly with student count, dominates primary list page.
4. **H12 register slug-collision retry path has `ReferenceError: slug is not defined`** (Medium, carryover from 2026-04-17 pen-test) — `src/routes/auth.js:228`. Gated behind `PUBLIC_REGISTRATION_ENABLED !== 'true'` so not exploitable in production, but still a latent crash if the flag ever flips or if the current branch is reached by any other entry.
5. **`rateLimit` middleware fails open for non-auth paths when D1 unreachable** (Medium, carryover M20) — `src/middleware/tenant.js:440-452`. `/api/contact`, `/api/signup`, `/api/covers`, `/api/auth/demo` all bypass rate-limiting during D1 outages; these are the exact endpoints that cost real money when abused.

The fundamentals remain strong: tenant isolation with pre-existence checks, parameterized SQL throughout, HS256-asserted JWTs with iss/aud/jti, AES-GCM encryption with fail-closed plaintext decryption, orgStatusCache via KV, zero console.log leaks in components, 1948/1948 passing tests, shared `PUBLIC_PATHS` constant preventing auth-bypass drift.

#### Medium — `rateLimit` middleware fails open for non-auth paths (M20 carryover)

**File**: `src/middleware/tenant.js:440-452`

```js
} catch (error) {
  console.warn('Rate limiting error:', error.message);
  const sensitiveAuthPaths = [
    '/api/auth/login',
    '/api/auth/register',
    '/api/auth/reset-password',
    '/api/auth/forgot-password',
  ];
  if (sensitiveAuthPaths.some((p) => c.req.path.startsWith(p))) {
    return c.json({ error: 'Service temporarily unavailable' }, 503);
  }
}
// falls through to next() for everything else
```

During any D1 outage or `rate_limits` table issue, `/api/contact` (5/min), `/api/signup` (5/min), `/api/covers` (60/min), and `/api/auth/demo` (10/min) all silently bypass rate-limiting. These are the exact endpoints that cost real money when abused — Resend quota, R2 writes, Worker CPU on cover lookups, demo JWT minting.

**Fix**: Extend the fail-closed list:
```js
const sensitiveAuthPaths = [
  '/api/auth/login', '/api/auth/register',
  '/api/auth/reset-password', '/api/auth/forgot-password',
  '/api/auth/demo',
  '/api/contact', '/api/signup',
  '/api/covers/',
];
```

#### Medium — `/api/auth/mode` still leaks infrastructure flags (L23 upgrade)

**File**: `src/routes/auth.js:112-128`

Endpoint is public and returns `{ d1Database: true, kvStorage: true }`. The `mode` and `ssoEnabled` fields are needed by the frontend to choose UI; the feature flags below are not consumed anywhere.

**Fix**: Drop `features` from the response (or omit the infrastructure booleans):
```js
return c.json({
  mode: isMultiTenant ? 'multitenant' : 'legacy',
  ssoEnabled: Boolean(c.env.MYLOGIN_CLIENT_ID),
});
```

#### Medium — `SELECT *` in user-delete handler reads password_hash unnecessarily

**File**: `src/routes/users.js:470,475`

```js
existingUser = await db
  .prepare('SELECT * FROM users WHERE id = ?')
  .bind(targetUserId)
  .first();
```

Only `role` and `organization_id` are consumed from the result. Fetching `password_hash` into memory violates least-privilege; a future refactor that logs/caches the row would leak hashes.

**Fix**: `SELECT id, role, organization_id FROM users WHERE id = ?`.

#### Low — `/api/health` exposes exact version (L22 carryover)

**File**: `src/worker.js:288-291`

Fingerprinting aid only; combined with GitHub releases, lets an attacker map installed version → known vuln. Leave it if /health is only hit by Cloudflare probes; redact the version if this endpoint is ever exposed to the public internet long-term.

### 2. Bugs & Data Integrity

#### High — Session POST mutates 6+ rows without transactional guarantees

**File**: `src/routes/students.js:1250-1406`

```js
studentsRouter.post('/:id/sessions', requireTeacher(), auditLog('create', 'session'), async (c) => {
  // … validation, book lookup, org scoping …
  await db.prepare(`INSERT INTO reading_sessions …`).run();           // 1308
  if (body.bookId)  await db.prepare(`UPDATE students SET current_book_id = ? …`).run();  // 1335
  if (!isMarkerSession) await db.prepare(`UPDATE students SET last_read_date = MAX(…)`).run();  // 1350
  const streakData   = await updateStudentStreak(db, id, organizationId, c.env);             // 1362
  await recalculateStats(db, id, organizationId);                                             // 1365
  const newBadges    = await evaluateRealTime(db, id, organizationId, student.year_group);    // 1368
  const completedGoals = await updateClassGoalOnSession(db, id, organizationId);              // 1373
```

No outer try/catch, no `db.batch()`. If any step after line 1331 throws (D1 transient, streak calc bug, badge evaluation error), the session INSERT already committed and the student's summary fields are stale. D1 lacks multi-statement transactions, but `db.batch([insert, update, update])` is atomic and would at least cover the first three writes. The rest (streak/stats/badges/goals) can be re-run idempotently by the next session.

**Fix**: Wrap the three core writes in a single `db.batch([...])`. Keep the downstream side-effects (streak/stats/badges/goals) outside the batch but inside an outer try/catch that logs to Sentry and returns 201 anyway — the session is created and the side-effects self-heal on next write or nightly cron.

#### High — `classGoalsEngine.recalculateClassGoalProgress()` runs 6 queries per goals GET

**File**: `src/routes/classes.js:429` → `src/utils/classGoalsEngine.js:144-246`

Every `GET /api/classes/:id/goals` call (teacher dashboard, classroom projection, admission-time checks) unconditionally recalculates progress, which fires 6 separate `SELECT COUNT … FROM reading_sessions` / `books` / `genres` / `reading_days` / `readers` / `student_badges` queries. No "dirty" check. The classroom projection view polls this while running, making it one of the hottest endpoints by query cost.

**Fix**: Two options, pick one:
1. **Skip recalc if unchanged**: store `last_session_at` on `class_goals`. If `MAX(reading_sessions.session_date) <= last_session_at`, skip recalc and return cached values.
2. **Collapse into one query**: use conditional aggregation (`COUNT(CASE WHEN … END)`) or 6 scalar subqueries in one SELECT — D1 supports both. One round-trip to compute everything.

Option 2 is simpler and removes the staleness risk.

#### Medium — `DataContext.reloadDataFromServer` has no AbortController on Promise.all

**File**: `src/contexts/DataContext.js:55-124`

```js
const reloadDataFromServer = useCallback(async () => {
  const [studentsResponse, classesResponse, ...] = await Promise.all([
    fetchWithAuth(`${API_URL}/students`),
    fetchWithAuth(`${API_URL}/classes`).catch(() => null),
    // … 3 more
  ]);
```

Rapid org switching (owner toolbar) or component unmount mid-reload calls `setStudents`/`setClasses` on an unmounted tree. React 19 tolerates this with a warning but data can arrive out of order (5 parallel fetches = 5 independent race outcomes). Same pattern that was flagged and fixed in `HomeReadingRegister`.

**Fix**: Create an `AbortController` inside `reloadDataFromServer`, pass `signal` to all 5 `fetchWithAuth` calls, and cancel on unmount or re-invocation.

#### Medium — `AchievementsPage` stats fetch lacks AbortController

**File**: `src/components/AchievementsPage.js:37-54`

```js
useEffect(() => {
  if (currentTab !== 1) return;
  setStatsLoading(true);
  fetchWithAuth(`/api/students/stats?${params}`)
    .then(r => r.ok ? r.json() : null)
    .then(data => { setStats(data); setStatsLoading(false); })
    .catch(() => { setStats(null); setStatsLoading(false); });

```

Rapid tab switching or class-filter changes stack setState calls on the wrong value. Matches the HomeReadingRegister/DataContext gap.

#### Medium — `orgPurge` still loops 26 DELETEs individually (carryover 2026-04-14)

**File**: `src/services/orgPurge.js:94-143`

Confirmed still present. See prior audit for fix (rewrite as single `db.batch`).

#### Medium — `classGoalsEngine` aggregation queries have no per-query error handling

**File**: `src/utils/classGoalsEngine.js:144-246`

Six sequential queries. If query #3 fails, queries #4–#6 run against partially-initialised locals and the goals row is updated with bogus counts. Wrap each aggregation in try/catch; on failure preserve the prior value rather than overwriting with 0.

#### Medium — Book-cover R2 put failures silently drop

**File**: `src/routes/covers.js` (both `/:type/:key` and `/search` branches)

`waitUntil(r2.put(...).catch(err => console.error))` — R2 quota or auth failures log to Cloudflare dashboard only. No Sentry breadcrumb. Subsequent requests re-fetch upstream (Google Books / Hardcover) and repeat the R2 miss.

**Fix**: `Sentry.captureException(err)` inside the `.catch`; include the ISBN/title key for correlation.

#### Medium — Register slug-collision retry path references undefined vars (H12 carryover)

**File**: `src/routes/auth.js:228`

```js
finalSlug = `${slug}-${slugCounter++}`;   // `slug` and `slugCounter` undefined
```

Gated behind `PUBLIC_REGISTRATION_ENABLED !== 'true'` so unreachable in prod. But: (a) if the flag is ever flipped, a slug collision will throw ReferenceError instead of retrying, (b) the intent of the code is retry-on-UNIQUE, so the bug hides a silent feature failure.

**Fix**: `finalSlug = \`${finalSlug}-${Math.random().toString(36).slice(2, 6)}\`;` — deterministic-ish suffix, no external state. Or delete the retry branch entirely since the feature is dormant.

Still present. See prior audit.

### 3. Performance & Scalability

#### High — `/api/students` uses SELECT * with correlated subqueries (carryover)

**File**: `src/routes/students.js:243-250`

Two correlated `(SELECT COUNT(*) … WHERE rs.student_id = s.id)` subqueries per row. At 1000 students this is 2000 extra subquery executions for the primary list page. Already flagged 2026-04-14; confirmed still present.

**Fix**: LEFT JOIN to pre-aggregated `student_reading_stats` + `student_badges` counts, avoid SELECT *.

#### High — AI recommendations endpoint has no server-side random-sample limit

**File**: `src/routes/books.js` — `generateBroadSuggestions` path

Performance agent flagged: catalog fetch + shuffle before sampling. At 18k books this is a large in-memory op per request. Indexed `ORDER BY RANDOM() LIMIT 100` directly in SQL is much cheaper.

**Fix**: Replace the fetch-all-then-shuffle with `SELECT ... ORDER BY RANDOM() LIMIT 100` against the org's filtered catalog.

#### Medium — Badge engine N+1: author-subquery loop (carryover 2026-04-14)

**File**: `src/utils/badgeEngine.js` — Series Finisher evaluator

One COUNT per distinct author per student. Still present. See prior audit.

#### Medium — Missing compound index on `reading_sessions(organization_id, student_id, session_date)` (carryover)

**File**: migration pending

Fix: `CREATE INDEX idx_sessions_org_student_date ON reading_sessions(organization_id, student_id, session_date DESC);`

#### Medium — Sentry browser replay + tracing at 10% default

**File**: `src/instrument.js:10-22`

```js
replaysSessionSampleRate: 0.1,
tracesSampleRate: 0.1,
```

At 1000 DAU that's ~100 replays/day plus session events. Replays are 2–5 MB each on cellular; tracing is generally cheap but compounds. For a school product where volunteers use iPad in tent-wifi conditions, every MB matters.

**Fix**: `replaysSessionSampleRate: 0.01` (errors stay at 100% via `replaysOnErrorSampleRate`), `tracesSampleRate: 0.02` for now. Revisit once you have a signal on per-class issue volume.

#### Medium — `jsPDF` and `react-joyride` imported eagerly at app startup

**Files**: `src/utils/statsExport.js:1`, `src/components/tour/TourProvider.js:2`

jsPDF is ~190 KB min; react-joyride ~50 KB. The PDF export is one click on the stats page; the tour is used once per user. Both should be lazy.

**Fix**:
```js
// statsExport.js callers
const { generateStatsPDF } = await import('../utils/statsExport.js');
// App root
const TourProvider = React.lazy(() => import('./components/tour/TourProvider'));
```

`npm audit` now reports 4 moderate (0 high/critical). All have fixes available. dompurify transitive; hono direct (<4.12.14); uuid <14.0.0 direct; postcss transitive. None are exploitable in the current usage pattern (SSR with hono/jsx is not used; dompurify comes via html2canvas/jspdf chain).
