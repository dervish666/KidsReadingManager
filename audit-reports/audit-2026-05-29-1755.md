## Findings by Category

### A. Bugs & Data Integrity

#### H1 — `last_read_date` DB triggers silently defeat the v3.64.3 "school-only" fix
**Severity: High** | **Files**: `migrations/0037_assessment_to_integer.sql:63–100` (live), `migrations/0004_reading_sessions.sql:51–90` (original), conflicts with `src/routes/students/sessions.js:250–259`

Three triggers recompute `last_read_date` over **every** session on insert/update/delete:
```sql
CREATE TRIGGER IF NOT EXISTS update_student_last_read_insert
AFTER INSERT ON reading_sessions
BEGIN
    UPDATE students
    SET last_read_date = (SELECT MAX(session_date) FROM reading_sessions WHERE student_id = NEW.student_id),
        updated_at = datetime('now')
    WHERE id = NEW.student_id;
END;
```
`grep "DROP TRIGGER"` across all 56 migrations finds only the FTS triggers dropped — **never these**. Commit `94babc0` (v3.64.3, "last read date tracks school sessions only") changed the app layer to set `last_read_date` only for `location='school'`, non-marker sessions. But the trigger fires *during the INSERT*, before/independent of the app's own UPDATE, and computes `MAX` over home sessions and ABSENT/NO_RECORD markers too.

**Reproduction**: Mark a child ABSENT today (a marker row is inserted with `notes='[ABSENT]…'`). The trigger sets `last_read_date = today` (the marker's date is the new MAX). The student card (`StudentCard.js`, `student.lastReadDate`) and "needs attention" logic now show the absent child as having just read.

**Fix**: New migration `DROP TRIGGER IF EXISTS update_student_last_read_insert/delete/update;`. The app code already maintains `last_read_date` correctly on all create/update/delete paths, so the triggers are now both redundant and wrong.

#### H2 — Parent-portal home sessions advance teacher-facing `last_read_date`
**Severity: High** (same root symptom as H1; independent contributor) | **File**: `src/routes/parent.js:268–278`
```js
// Always update last_read_date for home sessions
coreWrites.push(db.prepare(
  `UPDATE students SET last_read_date = MAX(COALESCE(last_read_date, ''), ?), updated_at = datetime('now')
     WHERE id = ? AND organization_id = ?`).bind(sessionDate, studentId, organizationId));
```
The parent endpoint only ever creates `location='home'` sessions yet unconditionally advances `last_read_date` — the exact thing v3.64.3 removed from the school path. A weekend home log makes Monday's register show "last read: Saturday", masking that the child hasn't read *at school*. (Even after fixing this line, H1's trigger reintroduces the symptom — both must be addressed together.)

**Fix**: Remove the unconditional `last_read_date` update here. If a "last read anywhere" indicator is wanted for parents, store it in a separate column.

#### M1 — QuickReadingView "+" custom-count button is permanently disabled (regression in v3.67.1)
**Severity: Medium** | **File**: `src/components/sessions/QuickReadingView.js:311`, depends on `HomeReadingRegister.js:301–312`
```js
const maxMultiple = unrecordedPrev + 1;   // unrecordedPrev counts NONE days in previousDays
...
<Button disabled={isRecording || maxMultiple < 5}>+</Button>
```
`previousDays` is always exactly **3** days (`for (let i = 3; i >= 1; i--)`), so `unrecordedPrev ≤ 3` and `maxMultiple ≤ 4`. `maxMultiple < 5` is therefore **always true** → the "+" button is disabled for every student, every render. The custom multi-day-count dialog (supports up to 14 days) is unreachable from Quick view; teachers can no longer log 5+ catch-up days there. Introduced by commit `c3d8a38`.

**Fix**: Align the threshold with the window — gate on `maxMultiple < 2`, or decouple the "+" dialog from the 3-day `previousDays` window so it can offer the full `multipleMaxDays` range.

#### M2 — Multi-day backfill double-counts a non-contiguous already-read day
**Severity: Medium** | **File**: `src/components/sessions/HomeReadingRegister.js:538–584` vs disable logic at `QuickReadingView.js:189–194`

The v3.67.1 disable guard computes `unrecordedPrev` by counting NONE days **anywhere** in the 3-day window, but `handleQuickRecord` writes sessions on **consecutive** days walking backward from `selectedDate` and only skips re-writing on *marker* days (`dayHasMarker`), not on already-READ days.

**Reproduction**: selectedDate=Thu, previousDays=[Mon,Tue,Wed] with Mon=NONE, **Tue=READ**, Wed=NONE → `unrecordedPrev=2`, `maxMultiple=3` → button "3" enabled. Pressing "3" loops i=0(Thu), i=1(Wed ok), i=2(**Tue already READ, not a marker**) → a **second** session is written on Tuesday, double-counting it in totals, stats, and class-goal session counts.

**Fix**: Either make `maxMultiple` count the *contiguous* run of unrecorded days immediately preceding `selectedDate` (stop at the first recorded day), or have the backfill loop skip days that already hold a non-marker session.

#### L1 — No unique constraint on active parent tokens; bulk-generate is check-then-insert
**Severity: Low** | **Files**: `migrations/0056_parent_access_tokens.sql`, `src/routes/parent.js:464–498`
Nothing prevents two active (`revoked_at IS NULL`) tokens for the same `(student_id, academic_year)`. `POST /generate/:classId` selects students-without-tokens then inserts with no transaction, so a concurrent single-student regenerate (e.g. QRCodeSheet auto-gen racing a teacher opening ParentQRButton) can create duplicates; `GET /token/student/:studentId` then `.first()`s an arbitrary one, so the print sheet and the single-QR dialog can disagree on "the" link. Low impact (both validate). **Fix**: partial unique index `ON parent_access_tokens(student_id, academic_year) WHERE revoked_at IS NULL`.

#### L2 — `stats.js` "this week" count has no upper bound
**Severity: Low** | **File**: `src/routes/students/stats.js:128`
`SUM(CASE WHEN rs.session_date >= startOfWeek …)` has no `< nextWeek` bound (unlike `last_week`, which has both). Future-dated sessions (not normally creatable) would land in "this week". Cosmetic.

### B. Security

#### H3 — Rate limiter keys on the concrete request path, defeating per-resource/per-IP limiting on all parameterized routes
**Severity: High** | **File**: `src/middleware/tenant.js:408` (routes: `parent.js:79,176,339`; `covers.js:16`)
```js
const endpoint = c.req.path;   // concrete pathname, e.g. /api/parent/<TOKEN> or /api/covers/isbn/<ISBN>
...
SELECT COUNT(*) FROM rate_limits WHERE key = ? AND endpoint = ? AND created_at > datetime('now', ? || ' seconds')
```
In Hono, `c.req.path` is the concrete URL path, **not** the matched route pattern (`c.req.routePath`). So every distinct token/ISBN produces a distinct `endpoint` bucket — the "60/min" cap becomes "60/min *per token*", i.e. no aggregate cap on enumeration or fan-out. Affects parent `/:token` (×3) and covers `/:type/:key`.

- **Parent**: mitigated by 128-bit CSPRNG token entropy (`helpers.js:34` — verified, brute force infeasible), so this is not a practical PII-enumeration break, but it removes the per-IP load/abuse ceiling.
- **Covers**: each distinct cover key gets a fresh 60/min bucket → an attacker can drive unbounded R2 reads and external provider calls (Google Books / Hardcover are quota-/rate-limited and could be drained).

**Fix**: key the limiter on the route pattern, not the concrete path — pass an explicit bucket name into `rateLimit(max, windowMs, bucket)` and use that constant for the `endpoint` column (most robust), or use `c.req.routePath`. Add an IP-only aggregate bucket for `/api/parent/*` so guessing/abuse is bounded independent of the token value.

#### M3 — Parent portal: long-lived token-in-URL exposes child PII; token stored in plaintext at rest
**Severity: Medium** (intrinsic to the design; worth hardening) | **File**: `src/routes/parent.js:48–73, 79–170`
A valid token returns the child's first name, current book, streak, last 30 sessions, and badge count with no second factor, valid for up to a full academic year until manually revoked. The token is the full credential in a printed/shared URL (QRCodeSheet) and is stored plaintext in `parent_access_tokens.token`. Mitigations present and verified: `Cache-Control: no-store` (`parent.js:162`), global `Referrer-Policy: strict-origin-when-cross-origin` (`worker.js:139`), teacher-revocable, immediate revocation (validated live each request). **Fix (defense-in-depth)**: store only a hash of the token; consider a shorter independent TTL with easy re-request; document the exposure model in the DPA.

#### L3 — Parent paths absent from `FAIL_CLOSED_PATHS` (fail-open on D1 outage)
**Severity: Low** | **File**: `src/middleware/tenant.js:15–24, 393–399`
If D1 is unreachable, `rateLimit()` calls `next()` for parent routes. Combined with H3 this means parent abuse is unthrottled during a D1 incident. **Fix**: add `/api/parent/` to `FAIL_CLOSED_PATHS`.

#### L4 — Student PUT writes `classId` without verifying the class belongs to the caller's org
**Severity: Low** | **File**: `src/routes/students.js` (PUT handler, ~`:367`)
A teacher could assign a student to a guessed foreign `class_id`. Impact is limited (all class reads are org-scoped, so the foreign class never renders), but it stores a cross-tenant reference. **Fix**: validate `classId` against `classes WHERE id=? AND organization_id=?` before update.

### C. Compliance & Child Safety (UK GDPR / KCSIE)

#### H4 — Special-category pupil data returned to the `readonly` role on the bulk list endpoint
**Severity: High** | **Files**: `src/utils/rowMappers.js:63–72`, `src/routes/students.js:66`
`rowToStudent` unconditionally emits `senStatus, pupilPremium, ealStatus, fsm, dateOfBirth, gender, firstLanguage, ealDetailedStatus`. `GET /api/students` (the bulk list) is guarded only by `requireReadonly()`. Per CLAUDE.md, MyLogin maps **student → readonly**, so a student-role account can pull every classmate's SEN/FSM/pupil-premium/DOB in one call (visible via the API regardless of what the UI renders).

Regulation: UK GDPR Art. 5(1)(c) data minimisation; Art. 9 special-category data (SEN/health; FSM/PP/EAL as socioeconomic/ethnic proxies); KCSIE need-to-know. None of these fields is required to render a reading list.

**Fix**: add `rowToStudentMinimal()` for the list/readonly path stripping protected characteristics; gate the sensitive fields behind teacher/admin and the single-student detail view; confirm `readonly` accounts cannot reach them at all.

#### M4 — Parent GET view ignores the Article 18 processing restriction the POST honours
**Severity: Medium** | **File**: `src/routes/parent.js:79–170` (cf. POST guard at `:188–193`)
`validateParentToken` selects `processing_restricted` (`:58`) and the POST blocks restricted students, but `GET /api/parent/:token` has no such guard — a restricted child's reading record is still served. Under Art. 18, restricted data may only be stored; disclosure is further processing. Severity tempered (Medium) because the token holder is the child's own parent and restrictions are rare, but the asymmetry with the POST is clearly unintended. **Fix**: mirror the POST guard at the top of the GET.

#### M5 — No zero-retention / no-training signal to any AI provider; sub-processor DPA gap
**Severity: Medium** | **File**: `src/services/aiService.js:136–283`
Child-derived data (reading level, genre prefs, liked/disliked titles, recent reads) is sent to Anthropic/OpenAI/Google with no ZDR/no-train flag (Anthropic call sets only `anthropic-version`; OpenAI sends no `store:false`; Google uses the consumer `generativelanguage.googleapis.com`, not Vertex). Standard tiers retain inputs ~30 days. **Positive verified**: `toAISafeProfile` strips DOB/age/gender/SEN/EAL/year-group/notes before any AI call (`studentProfile.js:195–218`) — demographic exfiltration is well-guarded. **Fix**: pursue Anthropic ZDR; set OpenAI `store:false`/enterprise; move Google to Vertex; maintain Art. 28 DPAs + UK IDTA/SCC documentation.

#### M6 — Content moderation enforced only on cache write, bypassed on cache hit
**Severity: Medium** | **File**: `src/routes/books/recommendations.js:341–381` vs `:532`
`filterContentSafe` runs on freshly generated suggestions and only moderated output is cached — but the cache-hit branch returns cached suggestions with no moderation pass. Safe today, fragile invariant: a denylist update, a pre-moderation cache entry, or any future path caching raw output would surface unfiltered content to a child. **Fix**: run the (cheap, synchronous) `filterContentSafe` on the cache-hit branch too.

#### M7 — MyLogin callback logs children's email and name in plaintext
**Severity: Medium** | **File**: `src/routes/mylogin.js:196, 208, 244, 282`
`console.error('[MyLogin] … - user:', email)` and `console.log(… for ${name})` — for student-role logins these are children's names/emails in Worker logs (auto-captured by the Cloudflare Sentry SDK). Browser-side `scrubSentryEvent` does not cover server logs. **Note**: the parent route logs are clean (opaque `studentId` UUID only, no token/name). **Fix**: log `mylogin_id`/hashed id instead of email/name; add a server-side Sentry scrubbing rule.

#### L5 — User erasure leaves a dangling `parent_access_tokens.created_by` reference
**Severity: Low** | **File**: `src/routes/users/gdpr.js` (erase batch)
User erasure deletes refresh/reset tokens but does not null/reassign `parent_access_tokens.created_by` (FK to `users.id`). D1 doesn't enforce FKs by default, so the delete succeeds but the erased teacher's id lingers in a child-data table. **Fix**: add `UPDATE parent_access_tokens SET created_by = NULL WHERE created_by = ?` to the erase. (Also: the `orgPurge.js` "26 tables" comment is stale — code is correct.)

#### L6 — Legacy `buildPrompt()` still concatenates raw user input (latent injection)
**Severity: Low** | **File**: `src/services/aiService.js:62–113`
The legacy library-constrained `buildPrompt` interpolates `likes`/`dislikes` raw, without `tagUserInput`. It appears dead (the live route uses `generateBroadSuggestionsWithFailover`), but if any path calls `generateRecommendations` the 2026-05-08 injection is live again. **Fix**: confirm no callers via grep, then delete; or apply `tagUserInput` for consistency.

### D. Performance, Cost & Quotas (Cloudflare)

#### H5 — Session-write side-effect fan-out, amplified by the register's client-side POST loop
**Severity: High** (scaling/cost; correct today) | **Files**: `src/routes/parent.js:280–319`, `src/routes/students/sessions.js:261–302`, `src/components/sessions/HomeReadingRegister.js:499–743`
Each session POST runs `updateStudentStreak` → `recalculateStats` → `evaluateRealTime` → `updateClassGoalOnSession` sequentially after the insert, ≈ 25–35 D1 subrequests per single tap. The frontend amplifies this: `handleQuickRecord` fires these POSTs in a sequential `for` loop — up to 14 for a multi-day entry, and the marker-day branch fires **two** POSTs per iteration (`HomeReadingRegister.js:555–572`). A 14-day catch-up = up to 28 sequential POSTs × ~20+ D1 queries each. A Sunday-evening burst of parent logging (10k students) ≈ 300k D1 queries. **Fix**: move side-effects into `ctx.waitUntil()` (return the 201 immediately; the nightly cron already reconciles), and/or add a true bulk session-create endpoint so the register stops looping POSTs.

#### M8 — `updateClassGoalOnSession` rescans the full academic year on every session write
**Severity: Medium** | **File**: `src/utils/classGoalsEngine.js:131–240, 254–313`
Every parent log and teacher session POST runs a 6-subquery aggregate scanning all of a class's reading_sessions for the whole academic year (one subquery does `JOIN json_each(b.genre_ids)`). For a 30-pupil class with a year of data that's ~27k row reads per save — when the nightly drift cron already reconciles goals. **Fix**: make the per-session update incremental, or drop it and rely on nightly reconciliation + optimistic client increment.

#### M9 — Org-list 4 correlated subqueries per row (carryover, STILL PRESENT)
**Severity: Medium** | **File**: `src/routes/organization.js:179–196`
4 correlated subqueries per returned row (student_count, class_count, last_sync_error, has_ai_key); the `hasErrors` filter adds a 5th correlated `EXISTS` evaluated pre-pagination across all orgs. Owner-only, so small blast radius, but unindexed counts scan child tables per org. **Fix**: replace with `LEFT JOIN (… GROUP BY organization_id)` derived tables; index `students(organization_id, is_active)`, `classes(organization_id, is_active)`, `wonde_sync_log(organization_id, started_at)`, `org_ai_config(organization_id)`.

#### L7 — Demo-reset badge phase is sequential and un-hoisted
**Severity: Low** (bounded to 24 demo students) | **File**: `src/services/demoReset.js:206–230`
Per-student `recalculateStats` + `evaluateRealTime` + `evaluateBatch`, none receiving a hoisted genre map (~360 sequential D1 queries/hour). **Fix**: hoist the genre map; use the already-batched `processBadgesForOrg` instead of the three legacy functions.

#### L8 — `rate_limits` cleanup is probabilistic; OpenLibrary parent search uncached; slug-in-loop on wonde sync-all
**Severity: Low** | Files: `tenant.js:449–459`, `parent.js:422–438`, `wondeAdmin.js:135`
(a) Cleanup is `Math.random() < 0.01` per write plus the daily cron — between runs the table holds up to a day of every public request (covers insert a row on *every* request). Confirm a composite index on `rate_limits(key, endpoint, created_at)`; consider KV-with-TTL for public counters. (b) Parent book search hits OpenLibrary per query with no cache. (c) `generateUniqueSlug` is awaited inside the wonde sync-all build loop (N sequential queries before the batched inserts). All low-frequency/owner paths.
