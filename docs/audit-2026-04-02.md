# Codebase Audit Report — Tally Reading
## Date: 2026-04-02
## Scope: Current codebase audit across security, tenancy, performance, correctness, and maintainability

## Executive Summary

Tally Reading has solid fundamentals: parameterized SQL is used consistently, tenant/auth middleware is centralized, refresh tokens are hashed and rotated, API responses get sensible security headers, and targeted security middleware tests are passing.

The biggest current risks are architectural rather than obvious single-line bugs:

1. The SPA stores access tokens in `localStorage` while the frontend is served without a Content Security Policy, which makes any future XSS bug much more damaging.
2. AI suggestion "in library" checks are not organization-scoped, so results can be wrong and can leak the existence of books outside the caller's selected library.
3. Several book endpoints and UI flows still assume "download a lot and filter client-side", which will become the main scaling bottleneck as libraries grow.

I also found a handful of medium-risk operational and code-quality issues: unbounded book query sizes, unthrottled external search calls, `ORDER BY RANDOM()` on large book tables, duplicate webhook slug collisions, and a small secret-handling hardening opportunity in the MyLogin token exchange.

## What I Reviewed

- Worker bootstrap, middleware chain, security headers, cron handling
- Auth, MyLogin SSO, refresh/logout/reset flows
- Tenant isolation and role enforcement
- Book, student, class, metadata, webhook, support, billing, and organization routes
- D1/KV data providers and recommendation-related query paths
- Auth/Data/UI React contexts and large-data frontend flows
- Existing audit docs and targeted security/unit middleware tests

## Positive Findings

- Tenant and RBAC enforcement are centralized in middleware, which reduces route-level drift.
- Refresh tokens are hashed in storage, rotated on use, and protected with reuse detection.
- API CORS is explicit and same-origin-first.
- SQL injection posture is generally good; I did not find raw user input interpolated into SQL values.
- Security-focused test coverage is better than average for a project of this size.

## Findings

### High

#### H1. Access tokens are persisted in `localStorage`, and the SPA is served without a CSP
- Evidence:
  - [src/contexts/AuthContext.js](/Users/dervish/CascadeProjects/TallyReading/src/contexts/AuthContext.js#L466) stores the access token in `window.localStorage`.
  - [src/contexts/AuthContext.js](/Users/dervish/CascadeProjects/TallyReading/src/contexts/AuthContext.js#L469) stores serialized user/org data in `localStorage`.
  - [src/worker.js](/Users/dervish/CascadeProjects/TallyReading/src/worker.js#L375) sets some security headers for static assets, but no `Content-Security-Policy`.
- Impact:
  - Today, a single XSS anywhere in the React app would expose bearer tokens and user context immediately.
  - Because access tokens authorize API requests directly, this materially increases account-compromise impact.
- Recommendation:
  - Move access-token handling to memory-only client state and rely on the existing httpOnly refresh cookie for renewal.
  - Add a strict CSP for frontend responses, ideally nonced if inline scripts are unavoidable.
  - Treat this as the highest-priority hardening item.

#### H2. AI suggestion "in library" checks are not organization-scoped
- Evidence:
  - Cached suggestion cross-check: [src/routes/books.js](/Users/dervish/CascadeProjects/TallyReading/src/routes/books.js#L495)
  - Fresh suggestion cross-check: [src/routes/books.js](/Users/dervish/CascadeProjects/TallyReading/src/routes/books.js#L572)
  - Both queries read from `books` only, not `org_book_selections`.
- Impact:
  - Suggestions can be marked `inLibrary: true` because a title exists globally, even when the current organization has not selected that book.
  - This is primarily a correctness bug, but in a multi-tenant product it also leaks global catalog presence across tenant boundaries.
- Recommendation:
  - Join `org_book_selections` and filter by `organization_id` and `is_available = 1`.
  - Add an integration test that proves org A does not see org B-only titles as local matches.

### Medium

#### M1. Book list/search endpoints accept unbounded page sizes and limits
- Evidence:
  - Paginated books route: [src/routes/books.js](/Users/dervish/CascadeProjects/TallyReading/src/routes/books.js#L100)
  - Book search route: [src/routes/books.js](/Users/dervish/CascadeProjects/TallyReading/src/routes/books.js#L171)
- Impact:
  - Very large `pageSize` or `limit` values can trigger expensive queries and oversized responses.
  - This is an avoidable abuse/performance risk, especially because these endpoints are reachable by any authenticated user.
- Recommendation:
  - Clamp `pageSize` and `limit` server-side, for example `1-100`.
  - Reject negative values instead of letting SQLite interpret them.

#### M2. External OpenLibrary search is not rate-limited
- Evidence:
  - The route performs live external fetches at [src/routes/books.js](/Users/dervish/CascadeProjects/TallyReading/src/routes/books.js#L676).
  - Worker-wide cost rate limits cover AI, metadata, and Hardcover only, not external title search: [src/worker.js](/Users/dervish/CascadeProjects/TallyReading/src/worker.js#L229).
- Impact:
  - Authenticated users can generate a large number of external requests through typeahead.
  - That increases latency variance and creates an easy abuse path against a third-party service.
- Recommendation:
  - Put `/api/books/search-external` behind a modest rate limit.
  - Consider short-lived caching by normalized query string.

#### M3. The frontend eagerly downloads large book datasets and filters them in the browser
- Evidence:
  - Global load fetches all minimal books on auth load: [src/contexts/DataContext.js](/Users/dervish/CascadeProjects/TallyReading/src/contexts/DataContext.js#L65)
  - `BookManager` then fetches the full book dataset again: [src/components/books/BookManager.js](/Users/dervish/CascadeProjects/TallyReading/src/components/books/BookManager.js#L54)
  - `BookAutocomplete` filters and sorts the local book array on each search term update: [src/components/sessions/BookAutocomplete.js](/Users/dervish/CascadeProjects/TallyReading/src/components/sessions/BookAutocomplete.js#L217)
- Impact:
  - Startup cost grows with library size.
  - Book-heavy screens pay twice: once for the global minimal list and again for the full list.
  - Client-side filtering and sorting become the dominant UI bottleneck for larger schools.
- Recommendation:
  - Replace eager full-library hydration with server-backed search and paginated fetches.
  - Keep only small caches client-side: recent books, top matches, or current page.

#### M4. Recommendation queries use `ORDER BY RANDOM()` on the full books table
- Evidence:
  - Main recommendation query: [src/data/d1Provider.js](/Users/dervish/CascadeProjects/TallyReading/src/data/d1Provider.js#L565)
  - Fallback query: [src/data/d1Provider.js](/Users/dervish/CascadeProjects/TallyReading/src/data/d1Provider.js#L624)
- Impact:
  - `ORDER BY RANDOM()` scales poorly and becomes increasingly expensive as the catalog grows.
  - This is a classic database hotspot and can become user-visible under load.
- Recommendation:
  - Replace with a cheaper sampling strategy: precomputed buckets, random ID windows, or candidate subsets selected first via indexed predicates.

#### M5. Wonde `schoolApproved` webhook can fail on duplicate school-name slugs
- Evidence:
  - New org slug generation is a single sanitized transform with no uniqueness handling: [src/routes/webhooks.js](/Users/dervish/CascadeProjects/TallyReading/src/routes/webhooks.js#L96)
  - `organizations.slug` is unique in schema: [migrations/0002_organizations_users.sql](/Users/dervish/CascadeProjects/TallyReading/migrations/0002_organizations_users.sql#L9)
- Impact:
  - Two schools with the same or similarly normalized name can cause approval to fail with a unique-constraint error.
  - This is an operational onboarding bug, not just a cosmetic issue.
- Recommendation:
  - Reuse the slug-collision handling pattern already present in auth registration.
  - Add a webhook test covering same-name school approvals.

#### M6. MyLogin token exchange sends the client secret twice
- Evidence:
  - Secret is included in both Basic auth and form body at [src/routes/mylogin.js](/Users/dervish/CascadeProjects/TallyReading/src/routes/mylogin.js#L125).
- Impact:
  - This is not an immediate vulnerability, but it increases accidental exposure surface in logs, proxies, or upstream diagnostics.
- Recommendation:
  - Prefer one standards-compliant mechanism unless MyLogin explicitly requires both.
  - If dual-send is kept for compatibility, document why.

### Low

#### L1. Webhook tests still attempt real outbound Wonde requests
- Evidence:
  - During `src/__tests__/unit/webhooks.test.js`, the suite attempted DNS resolution for `api.wonde.com`.
- Impact:
  - Tests remain non-hermetic and can become flaky in CI or offline environments.
  - This also makes failures noisier and slows investigation.
- Recommendation:
  - Mock `fetchSchoolDetails` or the underlying fetch in webhook tests.

#### L2. Encryption still falls back to `JWT_SECRET`
- Evidence:
  - [src/utils/crypto.js](/Users/dervish/CascadeProjects/TallyReading/src/utils/crypto.js#L530) uses `ENCRYPTION_KEY || JWT_SECRET`.
- Impact:
  - A dedicated key is preferred because compromise of the JWT secret also compromises encrypted config secrets.
  - The code comments already recognize this, so this is more of a hardening gap than a bug.
- Recommendation:
  - Treat `ENCRYPTION_KEY` as required in production and plan a gradual re-encryption migration.

## Suggested Priority Order

1. Remove access-token persistence from `localStorage` and add a frontend CSP.
2. Fix org scoping for AI/library match checks.
3. Clamp all book list/search limits and rate-limit external title search.
4. Reduce eager full-library downloads and move autocomplete toward server-driven search.
5. Replace `ORDER BY RANDOM()` in recommendation queries.
6. Harden webhook slug creation and tidy up MyLogin secret handling.
